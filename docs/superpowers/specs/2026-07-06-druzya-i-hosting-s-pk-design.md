# Дизайн: Друзья и хостинг сервера с ПК

Дата: 2026-07-06
Проект: Domino Craft (`MinecraftLauncher`)
Статус: одобрен пользователем, готов к плану реализации

## Цель

Превратить лаунчер из «запускалки» в социальный инструмент для компании друзей:

1. **Друзья** — локальный список друзей и индикатор «кто сейчас онлайн» на общих серверах.
2. **Хостинг с ПК** — поднять сервер с модпаком прямо из лаунчера в один клик для
   спонтанных коротких сессий, с публичным адресом через туннель playit.gg.

Обе фичи дополняют, а не заменяют платный always-on сервер (он остаётся основным).

## Границы (scope) и ключевые решения

- **Без бэкенда.** Никакого отдельного сервиса/БД. Список друзей хранится локально,
  presence работает пингом известных серверов.
- **Друзья — односторонний локальный список.** Настоящих взаимных заявок «отправить/принять»
  и пуш-уведомлений НЕТ (они требуют общего хранилища). Пользователь сознательно выбрал простоту.
- **Идентификация — по нику** (вход в игру офлайн, по нику).
- **Хостинг с ПК = тот же модпак**, что и `manifest.json` (версия + Forge/Fabric + моды),
  иначе клиент друга не совпадёт с сервером.
- **Подключение к серверу с ПК — туннель playit.gg** (друзьям ничего настраивать не надо).
- **Адрес поднятого сервера пользователь копирует друзьям в Discord руками** — автоматической
  раздачи адреса нет (нет бэкенда/канала пушей).
- **Два сервера:** платный always-on (основной) + ad-hoc с ПК (казуальный).

### Реализация фазами

- **Фаза 1 — Друзья.** Маленькая, самостоятельная, опирается на готовый `status.js`.
- **Фаза 2 — Хостинг с ПК.** Крупнее; переиспользует загрузчики Java/лоадеров/модов
  (Фаза 0, уже в лаунчере) и вайтлист из списка друзей (Фаза 1).

Порядок: сначала Фаза 1, потом Фаза 2 — так удобнее и по зависимостям, и по видимому результату.

## Архитектура

Вписывается в текущую структуру `main/` (Node) + `renderer/` (UI) + `preload.js` (IPC-мост).
Бэкенда нет.

| Модуль | Назначение | Зависит от |
|--------|-----------|-----------|
| `main/friends.js` | Локальный список друзей (JSON в `userData`): добавить/удалить/сохранить | — |
| `main/presence.js` | Периодический пинг серверов, сверка ников → «кто онлайн и где» | `status.js`, `friends.js` |
| `main/host-server.js` | Поднять серверную сторону модпака, старт/стоп, стрим лога | `downloader.js`, `java.js`, `loaders.js`, `manifest.js` |
| `main/tunnel.js` | Управление агентом playit.gg, получение публичного адреса | — |
| `renderer/` | Панель «Друзья» и секция «Хостить с ПК» | preload/IPC |

## Фаза 1 — Друзья

### Модель данных

`userData/friends.json`:

```json
{
  "friends": [
    { "nick": "Petya", "note": "сосед", "addedAt": 1720000000000 }
  ],
  "watchedServers": [
    { "label": "Наш сервер", "host": "play.example.com", "port": 25565 }
  ]
}
```

- `friends` — ники. Добавление одностороннее: вписал ник → сохранилось.
- `watchedServers` — какие серверы пинговать для presence. Платный сервер добавляется
  автоматически из манифеста (`manifest.server`); можно вручную добавить временный адрес
  чьего-то ad-hoc сервера, чтобы видеть друзей и там.

### Логика presence (`main/presence.js`)

- Раз в ~15 сек пингует каждый `watchedServer` через существующий `getServerStatus(host, port)`.
- Из ответа берёт список игроков онлайн, сверяет с `friends` по нику.
- Отдаёт в UI массив `{ nick, online, server }`.
- Пинги параллельны, с таймаутом (в `status.js` уже 3 сек). Недоступный сервер = игроки на нём
  «оффлайн», без падения всего списка.

**Ограничение (заложено в дизайн):** ответ сервера отдаёт не всех игроков, а выборку (sample) —
обычно до ~12 ников, и админ может её отключить. Если sample пустой/урезан — показываем только
«на сервере N игроков» без поимённого списка. На малолюдном сервере (<12) это не проблема;
на платном sample можно принудительно не резать настройкой сервера.

### UI — панель «Друзья»

- Список друзей с индикатором ● онлайн (и на каком сервере) / ○ оффлайн.
- Поле «добавить по нику» + удаление друга.
- Кнопка «копировать адрес сервера» рядом с онлайн-другом (быстро зайти туда же).

### IPC

`friends:list`, `friends:add`, `friends:remove`, `friends:watch-add`, `friends:watch-remove`.
Presence лаунчер шлёт в renderer событием `presence:update` по таймеру.

## Фаза 2 — Хостинг с ПК

### `main/host-server.js` — подготовка и запуск серверной стороны

Рабочая папка: `userData/host-server/` (мир там же — сессии продолжаются).

1. **Файлы сервера:**
   - серверный jar нужной версии Minecraft (из манифеста Mojang);
   - серверная часть Forge/Fabric (переиспользуем логику `loaders.js`, серверный вариант);
   - моды из манифеста модпака (те же файлы, что качает клиент);
   - `server.properties` с `online-mode=false`, `motd`, `max-players`, порт 25565;
   - `eula.txt`.
2. **EULA:** при первом запуске показываем окно «принять EULA Mojang». Молча ставить `eula=true`
   нельзя — нужно явное согласие пользователя. Приняли → пишем файл, дальше не спрашиваем.
3. **Запуск:** `java ... -jar server.jar nogui` дочерним процессом; стримим stdout в UI;
   ловим строку готовности `Done (...)! For help...` → статус «сервер готов».
4. **Управление:** старт / стоп. Стоп — мягко: команда `stop` в stdin, при таймауте — принудительно
   (чтобы не потерять мир). Статусы: `preparing → starting → ready → stopping`.

### Защита и авторизация

- `online-mode=false` (вход по нику), поэтому обязателен **вайтлист** (`white-list=true`).
  Вайтлист **автоматически формируется из списка друзей** (Фаза 1).
- Опционально — auth-мод с паролем (для Fabric — EasyAuth или аналог для Forge).

### Память

**Отдельный ползунок памяти для сервера**, независимый от памяти игры (хостить и играть на одном ПК
тяжело). В UI — подсказка «рекомендуется мощный ПК / проводной интернет».

### `main/tunnel.js` — playit.gg

- Лаунчер несёт агент playit бандлом.
- **Первый запуск:** агент выдаёт ссылку-claim → показываем её пользователю, он один раз привязывает
  туннель к своему playit-аккаунту. Дальше адрес постоянный.
- Агент направлен на `localhost:<порт сервера>`; из вывода парсим публичный адрес `xxxxx.playit.gg`
  → показываем с кнопкой «копировать».
- Туннель поднимаем ПОСЛЕ готовности сервера, гасим вместе с ним.

### UI — секция «Хостить с ПК»

- Большая кнопка Старт/Стоп.
- Статус (`готовлю → запускаю → готов`).
- Публичный адрес с кнопкой копирования.
- Счётчик игроков.
- Свёрнутый хвост лога.
- Ползунок памяти сервера.

## Обработка ошибок

- Java/сервер не скачался или упал при старте → показываем причину из лога, UI не виснет.
- playit не привязан / нет сети → статус «туннель не поднялся», сервер жив (можно играть по LAN).
- Порт 25565 занят → берём следующий свободный, передаём его в туннель.
- Пинг сервера в presence не прошёл → друг помечается оффлайн, остальной список работает.
- Остановка сервера с таймаутом → принудительное завершение, но сначала мягкий `stop`.

## Тестирование (vitest, в стиле существующих 42 тестов)

**Фаза 1:**
- `friends.js`: добавление, дедупликация, удаление, сохранение/загрузка JSON.
- `presence.js`: по моку ответа сервера + списку друзей → корректный «кто онлайн»; урезанный sample
  → «N игроков» без имён; недоступный сервер → оффлайн без падения.

**Фаза 2:**
- Генерация `server.properties` / `eula.txt` из конфига.
- Парсинг адреса playit из строки лога.
- Детект строки готовности сервера.
- Логика старт/стоп с замоканным `child_process`.

Сеть и `child_process` — замоканы.

## Что НЕ входит (YAGNI)

- Взаимные заявки в друзья, пуш-уведомления, общий бэкенд.
- Автоматическая раздача адреса ad-hoc сервера друзьям.
- Интеграция с Discord-ботом (рассматривалась, отклонена в пользу простоты; можно вернуться позже).
- Голосовой чат, кастомные права, панель администрирования сервера.
