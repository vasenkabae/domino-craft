# Дизайн: смена скина прямо в лаунчере

Дата: 2026-07-11
Проект: MinecraftLauncher (UI) + новый серверный плагин DominoSkins + бэкенд launcher_link.py на VDS
Статус: одобрен пользователем, готов к плану реализации

## Цель

Дать игроку сменить скин, не выходя из лаунчера Domino Craft — загрузил PNG, применил, при следующем
входе на сервер все видят новый скин. Важное ограничение контекста: сервер офлайн-режима (см.
`offline-auth.js` — UUID v3 от `OfflinePlayer:<ник>`, не настоящие Microsoft-аккаунты), поэтому
vanilla-клиенты не могут просто получить скин с серверов Mojang по UUID — там его нет. Требуется
собственный механизм раздачи и применения скина.

## Ключевые решения (из обсуждения)

- **Аудитория:** скин должен быть виден всем игрокам, включая vanilla-клиентов без Fabric-мода —
  не только тем, кто зашёл через лаунчер с DominoCitiesUI.
- **Источник картинки (v1):** загрузка PNG-файла с диска. Ник премиум-аккаунта Mojang — не в scope v1.
- **Момент применения:** скин обновляется **при следующем входе на сервер**, не мгновенно для уже
  зашедших игроков. Это осознанное упрощение — не нужно рассылать обновлённый профиль всем онлайн-
  клиентам.
- **Превью:** в лаунчере показываем 2D-превью скина (перёд/зад) перед применением.
- **Подход к рендерингу — вариант A, без сторонних зависимостей:** используем нативный Paper API
  (`PlayerProfile.getTextures().setSkin(url)`, доступен с Paper 1.18.2+) — назначаем скин по прямой
  ссылке на PNG, без похода за подписью к Mojang. Это неофициально задокументированное поведение,
  требует проверки на живом сервере (аналогично тому, как эмпирически подбиралась высота посадки в
  `/sit`-плагине DominoSit).
  - **Фоллбэк**, если вариант A не сработает на vanilla-клиентах: SkinsRestorer + mineskin.org для
    подписи скина. Тяжелее (сторонний Java-плагин + внешний API с лимитами), но отраслевой стандарт
    для офлайн-серверов. Переходим на него только если тестирование покажет, что unsigned-текстуры
    не рендерятся у части клиентов.
- **Авторизация загрузки:** без отдельной аутентификации сверх ника — соответствует существующей
  доверительной модели офлайн-сервера (`offline-auth.js`): кто ввёл ник в лаунчере, тот им и
  распоряжается. Осознанное упрощение, не баг.

## Архитектура

Три части, каждая — расширение уже существующего компонента проекта (без новых процессов/портов на
и так небольшом VDS — 1 core/1GB RAM):

1. **Лаунчер (Electron/Node.js)** — новый экран: выбор PNG-файла (64×64), 2D-превью перёд/зад
   (рисуем через `<canvas>` по стандартной развёртке скина), кнопка «Применить». Шлёт
   `multipart/form-data` на бэкенд.
2. **Бэкенд на VDS** (`discord bot/cogs/launcher_link.py` — тот же процесс, что уже держит
   `add_static("/dc/", MODPACK_DIR)` и `/launcher/verify`) — новый роут `POST /launcher/skin
   {name, file}`. Сохраняет как `MODPACK_DIR/skins/<ник>.png`, раздаётся статикой на
   `http://138.16.181.96:8765/dc/skins/<ник>.png`.
3. **Серверный плагин DominoSkins** (новый, по образцу тестового `DominoSit` — либо отдельный
   плагин, либо модуль внутри `DominoCities`, решается на этапе плана) — на
   `AsyncPlayerPreLoginEvent` проверяет наличие `skins/<ник>.png` на раздаче и назначает через
   `player.getPlayerProfile().getTextures().setSkin(url)`.

## Компоненты

### Лаунчер — экран смены скина
- Диалог выбора файла, фильтр на `.png`.
- Клиентская валидация: расширение, лимит размера файла 512 КБ (64×64 PNG весит от силы
  десятки КБ — с большим запасом на метаданные).
- 2D-превью перёд/зад по загруженному файлу.
- Кнопка «Применить» → `POST /launcher/skin` (ник текущего профиля + файл).
- Отображение результата: успех или текст ошибки от бэкенда.

### Бэкенд — роут в launcher_link.py
- Принимает `multipart/form-data`.
- Валидация по содержимому файла (не по расширению): реальный PNG, размер ровно 64×64 (легаси
  64×32 и произвольные размеры — отклонять, для простоты и защиты от кривых файлов), лимит веса
  512 КБ (сервер — та же граница, что и в лаунчере, второй рубеж защиты).
- Санитизация ника перед использованием в имени файла — только `[a-zA-Z0-9_]` (как допустимые
  символы ников Minecraft), защита от path traversal.
- Сохраняет в `MODPACK_DIR/skins/<ник>.png`, отвечает `200 OK` или текстом ошибки валидации.

### Плагин DominoSkins
- `AsyncPlayerPreLoginEvent`: HTTP-запрос на `http://138.16.181.96:8765/dc/skins/<ник>.png` с
  коротким таймаутом (~1–2 сек — не подвешивать вход игрока при недоступности VDS).
- Файл есть (200) → `setSkin(url)` на профиле игрока.
- Файла нет (404) или сетевая ошибка/таймаут → профиль не трогаем, игрок заходит со скином по
  умолчанию. Вход **никогда** не блокируется из-за этой фичи.

## Поток данных

Игрок открывает лаунчер → выбирает PNG → превью показывает развёртку → «Применить» → бэкенд
валидирует и сохраняет файл → лаунчер показывает подтверждение → при следующем входе на сервер
`DominoSkins` находит файл на раздаче и назначает скин → все игроки (включая vanilla) видят
обновлённый вид.

## Обработка ошибок

| Ситуация | Поведение |
|---|---|
| Файл не PNG / битый | Бэкенд отклоняет с понятным сообщением, лаунчер показывает его пользователю |
| Не 64×64 | Отклоняем с сообщением «нужен файл 64×64» |
| VDS недоступен при загрузке из лаунчера | Сетевая ошибка в UI лаунчера — «не удалось сохранить, попробуй позже» |
| VDS/файл скина недоступен в момент входа в игру | Плагин молча пропускает, вход не блокируется |
| Ник с недопустимыми символами | Санитизация на бэкенде отбрасывает лишнее до записи файла |

Приоритет обработки ошибок: фича скинов не должна ронять ничего постороннее — ни вход игроков, ни
раздачу модпака, ни стабильность Discord-бота (роут живёт в том же процессе, что и бот).

## Тестирование

Юнит-тестов на сервере минимум (как и в существующих плагинах DominoCities/DominoSit) — упор на
ручную проверку в игре, итеративно:
- Загрузка валидного PNG → скин появляется после реконнекта.
- Загрузка невалидного файла (не PNG, не 64×64) → лаунчер показывает ошибку, файл не сохраняется.
- VDS недоступен/таймаут в момент входа → вход всё равно проходит без зависаний.
- **Ключевая проверка гипотезы:** скин виден другому игроку на **ванильном клиенте без Fabric-мода**
  — если нет, откатываемся на фоллбэк (SkinsRestorer + mineskin.org).

## Что НЕ входит в v1 (YAGNI)

- Клонирование скина по нику существующего премиум-аккаунта Mojang.
- Мгновенное обновление скина для уже подключённых игроков без реконнекта.
- 3D-превью персонажа (только 2D перёд/зад).
- История/галерея сохранённых скинов — только один текущий скин на ник.
- Отдельная аутентификация загрузки сверх ника (соответствует модели офлайн-сервера).

## Фазы реализации

- **Фаза 1:** бэкенд-роут `/launcher/skin` в `launcher_link.py` + плагин `DominoSkins`
  (`setSkin(url)` при входе) — проверить гипотезу про unsigned-текстуры на живом сервере с
  vanilla-клиентом ДО того, как вкладываться в UI лаунчера.
- **Фаза 2:** экран в лаунчере (выбор файла, превью, отправка) — только после подтверждения, что
  Фаза 1 реально работает.

Порядок обязателен: если гипотеза варианта A не подтвердится, придётся менять подход на бэкенде
(SkinsRestorer) — не хочется переделывать готовый UI лаунчера под другой подход.
